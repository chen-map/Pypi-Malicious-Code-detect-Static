# PyPI恶意包静态分析系统 - 完整技术方案

## 项目概述

本项目是一个**国家级信息安全竞赛**的恶意包检测系统，核心功能是：
1. 对PyPI恶意包进行**静态分析**，构建完整的数据依赖图（DDG）
2. 通过**图分割算法**提取危险子图（包含恶意行为的代码片段）
3. 自动生成**可执行的桩程序**，交给动态分析团队进行沙箱测试

**设计目标**：至少80%生成的桩程序能够直接运行并正确触发恶意行为。

## 核心流程详解

### 阶段1: DDG构建 (Data Dependence Graph)

#### 1.1 目标
构建一个**项目级别的完整数据依赖图**，追踪所有变量的数据流向，识别危险操作的来源。

#### 1.2 核心算法: ddg_builder_v7.py

```python
# 文件: src/ddg_builder_v7.py
# 核心类: ProjectDDGBuilderV7

class ProjectDDGBuilderV7:
    def build(self):
        # 步骤1: 扫描所有Python文件
        py_files = self._scan_python_files()

        # 步骤2: 对每个文件构建DDG
        for file_path in py_files:
            self._build_file_ddg(file_path)

        # 步骤3: 跨文件数据流分析
        self._link_cross_file_edges()

        # 步骤4: 安全检测
        security_report = self._security_detection()

        return {
            'nodes': self.nodes,      # 全局节点表
            'edges': self.edges,      # 全局边表
            'nx_graph': nx_graph,     # NetworkX图对象
            'symbol_table': symbols,  # 符号表 (函数/类)
            'security_report': report
        }
```

### 阶段2: 图分割 (Graph Partitioning)

#### 2.1 为什么需要图分割？

**问题**：完整DDG可能包含数千个节点，难以直接分析。
- 大型项目的DDG可能超过10,000节点
- 动态分析团队需要**独立的、小规模的**测试用例
- 一个恶意包可能包含**多个独立的恶意行为**

**解决方案**：将大图分割为多个小的、独立的子图。

#### 2.2 三种分割算法

**方法1: WCC (Weakly Connected Components)**
```python
# 原理: 弱连通分量分割
# 适用: 提取完全独立的子图
subgraphs = list(nx.weakly_connected_components(nx_graph))
```

**方法2: BFS (Bidirectional Data Flow Truncation)**
```python
# 原理: 从危险节点双向BFS，限制最大节点数
def bfs_truncate(dangerous_node, max_nodes=500):
    # 向上追踪数据来源
    sources = bfs_ancestors(dangerous_node, max_depth=10)
    # 向下追踪数据使用
    uses = bfs_descendants(dangerous_node, max_depth=10)
    return combine(sources, uses)
```

**方法3: HYBRID (推荐)**
```python
# 策略: 先用WCC分割大分量，再用BFS截断超大子图
def hybrid_partition(nx_graph, max_nodes=500):
    # 步骤1: WCC初步分割
    components = list(nx.weakly_connected_components(nx_graph))

    # 步骤2: 检查每个分量大小
    subgraphs = []
    for comp in components:
        if len(comp) <= max_nodes:
            subgraphs.append(comp)
        else:
            # 步骤3: 超大分量用BFS二次分割
            subgraphs.extend(bfs_truncate(comp, max_nodes))

    return subgraphs
```

### 阶段3: 桩程序生成 (Stub Program Generation) ⭐核心

#### 3.1 桩程序是什么？

**定义**: 一个可执行的Python脚本，用于测试原恶意包的特定恶意行为。

**作用**:
1. 将原包当作库导入
2. 准备测试数据
3. 调用原包的函数（或直接执行代码片段）
4. 捕获异常，记录执行结果

**生成示例**:
```python
# test_ddg_results.py (自动生成)

# 2. 准备测试数据

    try:
        # 方案B: 调用原包函数
        result = malicious.dangerous_func(cmd)
        print(f"Result: {result}")
    except Exception as e:
        print(f"Caught: {e}")

# 4. 主入口
if __name__ == '__main__':
    test_subgraph_001()
```

def _generate_test_logic(dangerous_nodes, test_inputs):
    test_lines = []

    # 按模块分组 (避免重复import)
    module_groups = _group_nodes_by_module(dangerous_nodes)

    for module_name, nodes in module_groups.items():
        # Import模块
        test_lines.append(f"import {module_name}")

        for node in nodes[:5]:  # 最多测试5个节点
            func_name = node.get('function_name')
            class_name = node.get('class_name')
            is_complex = _is_complex_case(node)

            # ✅ 方案B: 调用原包函数
            if func_name and not is_complex:
                if class_name:
                    # 类方法: 需要实例化
                    obj = f"obj_{class_name.lower()}"
                    test_lines.append(f"{obj} = {module_name}.{class_name}()")
                    call_code = f"{obj}.{func_name}(...)"
                else:
                    # 模块函数
                    call_code = f"{module_name}.{func_name}(...)"

                test_lines.append(f"result = {call_code}")

            # ⚠️ 降级方案A: 直接执行代码
            else:
                code = node.get('code')
                test_lines.append(f"result = {code}")

    return '\n'.join(test_lines)

#### 3.3 复杂情况处理

某些场景下无法生成函数调用，需要降级为方案A：

**Lambda函数**
```python
# 代码: lambda x: eval(x)
# 问题: Lambda是匿名的，无法通过名字调用
# 解决: 标记为复杂情况，未来使用LLM处理
if '<lambda>' in code:
    is_complex = True
    reason = "lambda函数"
```

**装饰器**
```python
# 代码: @staticmethod / @classmethod
# 问题: 装饰器改变了函数的调用方式
# 解决: 标记为复杂情况
if '@' in code:
    is_complex = True
    reason = "装饰器"
```

**魔术方法**
```python
# 代码: __init__, __call__, __getattr__
# 问题: 魔术方法由Python解释器自动调用
# 解决: 标记为复杂情况
if func_name.startswith('__') and func_name.endswith('__'):
    is_complex = True
    reason = "魔术方法"
```
